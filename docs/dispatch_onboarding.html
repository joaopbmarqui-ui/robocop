Analyst onboarding

Welcome to Dispatch

Dispatch is the official server-side terminal interface for launching and supervising Impala jobs on the Hadoop Edge Node. This guide takes you from first setup to confident daily use.

Official name: Dispatch Kerberos required Jobs survive disconnects
Onboarding checklist progress 0 / 0

1. Welcome to Dispatch

Dispatch is a server-side text user interface (TUI) that runs on the Hadoop Edge Node. You start it from a directory that contains your SQL files, launch Impala jobs, watch their progress, inspect logs, review history, and browse Impala metadata—without depending on a local Windows GUI.

Purpose

  • Launch Impala jobs from SQL files, monthly templates, or existing tables
  • Supervise running and recent jobs from one cockpit
  • Inspect logs, history, and Impala metadata in the same session

Main benefits

  • Jobs continue after terminal disconnects (detached runner)
  • CSV results write uncompressed to your launch directory
  • Built-in Kerberos checks and a two-job concurrency cap
Official name

The official product name is Dispatch. Use the dispatch command. This guide refers to the product exclusively as Dispatch.

After completing this onboarding you will be able to

  • Complete analyst onboarding and verify the dispatch launcher
  • Authenticate with Kerberos and start the TUI
  • Create, monitor, cancel, and inspect jobs across Overview, View Logs, and History
  • Browse Impala schemas and tables, including catalog names that begin with digits

2. Prerequisites

Confirm these items before installation. Only requirements verified in the repository are listed.

Confirmed requirements

  • Jupyter already installed on your computer (assumed for this audience).
  • Access to the Hadoop Edge Node over SSH (documented port 2222), typically with RSA SecurID at the Enter PASSCODE: prompt.
  • Python 3.10 or 3.11 available on the Edge Node for the shared runtime (Release Operator concern; analysts do not install Python packages).
  • klist and impala-shell on PATH (onboarding fails if either is missing).
  • Writable private data root: /ads_storage/$USER (override with DISPATCH_DATA_ROOT if set).
  • Shared Dispatch tree already deployed by a Release Operator (typically /ads_storage/dispatch), including an active shared runtime.
  • At least one .sql file in the directory from which you launch Dispatch (needed for the New Job SQL picker).
To be confirmed
  • Exact UI steps to open a terminal from your JupyterLab / JupyterHub environment.
  • Whether your team’s primary launch path is SSH-only, a Jupyter terminal, or both.
  • Your organization’s VPN name and RSA SecurID provisioning process for new analysts.

3. Installation

Analysts do not run install.sh or pip. After the Release Operator activates the shared runtime, each analyst runs lightweight onboarding.

Open a terminal

  1. Connect to the Edge Node (documented example: ssh -p 2222 <user>@<edge-node>).
  2. If you work from Jupyter, open a terminal in that environment and connect to the same Edge Node session your team uses.
To be confirmed: exact menu path to open a terminal inside your Jupyter product (not documented in this repository).

Run analyst onboarding

Run onboarding directly from the deployed tree. Do not run it with source.

/ads_storage/dispatch/onboard.sh

Onboarding creates or repairs private state under /ads_storage/$USER/.dispatch, writes the dispatch launcher to ~/.local/bin/dispatch, and updates your shell profile. It needs no dependency bundle or personal venv and never runs pip.

Activate PATH in the current shell (if prompted)

If onboarding prints the following, copy and run it once in the current shell:

export PATH="$HOME/.local/bin:$PATH"

Otherwise, open a new SSH session so the updated profile is loaded.

Verify installation

which dispatch
cd /path/to/your/sql/files
dispatch
  • Success: onboarding prints Dispatch onboarding complete.
  • Success: which dispatch shows a path under ~/.local/bin.
  • Success: the TUI opens (Overview / dashboard).
Release Operator note (not required for analysts)

Once per Edge Node, the Release Operator activates the shared runtime:

cd /ads_storage/dispatch
DISPATCH_PYTHON_BIN=$(command -v python3.11) ./install.sh

Analysts should not run this unless they are also acting as Release Operator.

Installation troubleshooting (verified)

dispatch not found

Action: export PATH, then which dispatch. If still empty, rerun /ads_storage/dispatch/onboard.sh.

Shared runtime missing or invalid

Action: stop and contact the Release Operator. Do not run pip.

klist / impala-shell not found

Action: onboarding exits before creating private state. Restore Edge Node tooling on PATH, then rerun onboarding.

4. Authentication with kinit

Impala and related cluster tools rely on Kerberos. Dispatch checks your ticket before launching jobs and refreshes the displayed time-to-live (TTL) about every 60 seconds. Launch is refused when a ticket is missing or has less than five minutes remaining.

Obtain a ticket (SSH / interactive shell)

kinit
klist
  • Default realm used by Dispatch: CORP.MASTERCARD.ORG (override with DISPATCH_KRB5_REALM if set).
  • Success indicator: klist shows a valid, non-expired ticket; Dispatch sidebar shows a TTL such as KRB 7h 23m, not missing.

Inside Dispatch

  • On New Job, press K to run interactive kinit.
  • Or open the command palette with Ctrl+P and choose Refresh Kerberos (kinit).

Jupyter sessions

When Dispatch detects a Jupyter environment and the ticket is missing or under five minutes, a Kerberos sign-in modal appears (fields: EID, Windows password; button: Sign in). Cancel exits Dispatch with the message that Kerberos sign-in is required.

Expiration and re-authentication

  • Minimum launch TTL: 300 seconds (5 minutes).
  • Sidebar may highlight a low TTL when under about one hour.
  • Renew with kinit (shell) or K / command palette (TUI).
Authentication troubleshooting

Dispatch reports Kerberos problems

Run kinit, confirm with klist, then launch dispatch again.

Launch refused: ticket missing or TTL under 5 minutes

Press K on New Job (or run kinit in the shell) and retry.

Jupyter sign-in fails

Correct EID / Windows password and retry. Documented failure text may include authentication errors such as incorrect password.

5. Starting Dispatch

Launch sequence

  1. Authenticate with kinit (if needed).
  2. Change to the directory that contains your SQL files.
  3. Start Dispatch.
cd /path/to/your/sql/files
dispatch

Dispatch captures the launch-time working directory once. CSV destinations resolve relative to that directory for the entire session.

What you should expect

  • Window title Dispatch; subtitle includes your launch directory.
  • Overview (jobs cockpit) opens after startup checks.
  • Kerberos TTL appears in the sidebar.
  • Warning if the terminal is smaller than 80×24.
  • If private setup is missing: notification to run onboard.sh.

7. Overview tab

Overview is the supervision cockpit: status, recent jobs, a live log preview, and quick actions.

Visible sections

  • Status strip: Kerberos TTL, RUNNING n / 2, finished and failed counts for 7 days.
  • Jobs table (running first · last 7 days): columns ID, Source, Destination, State, Elapsed.
  • Filter (press /): filter by id, file, table, or state; Esc clears.
  • Detail pane (when height ≥ 30): short log preview for the selected job; hint · V for full logs.
  • Action bar: event trail; buttons New Job [N], View Logs [V], Cancel [C].

How to interpret states

StateMeaning
PendingBriefly between launch and runner start
RunningOrchestrator process is alive
SucceededOrchestrator exited successfully
FailedNon-zero exit; may show a classified error code
CancelledCancelled from the TUI

Concurrency cap: at most two jobs may be Pending or Running at once.

Shortcuts

N New Job · V View Logs · C Cancel · H History · B Browse · / Filter · J/K or arrows move selection · Enter open detail

When to use this tab

Start here every session to check Kerberos health, watch active jobs, cancel if needed, and jump into full logs.

  4. CSV output (when selected) is written uncompressed to the launch-time working directory.

9. View Logs tab

View Logs opens Job Detail for the job selected on Overview or History. It is the live-tail viewer for that job’s run.log.

How to access

  • Overview: select a row → V, View Logs [V], or Enter
  • History: select a row → Enter or View Logs [Enter]
  • Sidebar View Logs (requires a current selection)

Controls and features

  • Summary: State, Source, Destination, Started, Elapsed, CSV
  • Error banner on Failed jobs (classified codes when matched)
  • Streaming indicator: Streaming / Paused / Complete / Failed / Cancelled
  • Log search (/), follow toggle (Space/F), jump top/bottom (g/G)
  • Copy job ID (Y); Clone finished jobs (R); Cancel running/pending (C)

How logs relate to execution

The detached runner appends orchestrator output to run.log under the job directory. Overview shows a short preview; Job Detail streams up to the on-screen bound (200 lines). Application process logs also exist at ~/.dispatch/dispatch.log / the private Dispatch home log (not shown in the TUI).

Informational vs warnings vs errors

  • Job states encode outcome (Succeeded / Failed / Cancelled / Running).
  • Failed jobs may show classified codes: SYNTAX, TABLE_NOT_FOUND, MEMORY, AUTH, QUEUE — each with a suggested next step.
  • Log line styling dims timestamps/comments; it does not define a separate warning severity channel in the viewer.
  • Advisor findings (on Preview / launch gate) use severities error, warning, and info.
Simple log troubleshooting flow
  1. Confirm job State on Overview or Job Detail.
  2. If Failed, read the error banner code and suggestion.
  3. Search the log for the matching lines.
  4. For AUTH: renew Kerberos and retry (or clone).
  5. For SYNTAX: Preview SQL, fix the file, relaunch.
  6. For QUEUE / MEMORY: reduce scope or retry off-peak / ask platform team.

10. History tab

History lists finished jobs older than the Overview’s 7-day window so you can find prior executions.

Columns and controls

  • Title: Job History · finished more than 7 days ago
  • Columns: ID, Table/Target, State, Finished At
  • Search: filter by table, date, or job-id
  • Sort cycle (S): date → state → table
  • Pagination: [ / ]
  • Actions: Back [B], View Logs [Enter]

Persisted job data lives as on-disk manifests under /ads_storage/$USER/.dispatch/jobs/ (and related logs). History loads finished jobs outside the active window; it does not auto-refresh on an interval.

Cancel is not offered from History—open logs / Job Detail for a running job from Overview instead.

11. Browse tab

Browse Impala Metadata lists tables in a schema, shows sizes, describes columns, and can drop tables. It does not create or run jobs.

Selectors and actions

  • Schema input (default aa_enc)
  • Table name filter (default *; hint: type firstword* and load)
  • Load Tables [S] — runs Impala SHOW TABLES
  • Table columns: Sel, Name, Size, Type
  • Sizes load in the background via SHOW TABLE STATS (pending …, unknown —)
  • Describe [Enter] — column metadata in the right pane
  • Select All [A] / click Sel checkbox — multi-select for drop
  • Sort (O): name → size
  • Drop [D] — destructive; see warning below

Names that begin with numbers

Catalog schemas/tables whose names begin with digits are allowed in Browse. Dispatch backtick-quotes digit-leading identifiers in Impala SQL. New Job creation paths still require classic identifiers (letter or underscore first) for user-supplied names.

Warning: Drop is irreversible

Dropping tables requires typing exactly I AM SURE, then confirming DROP. The confirmation states that this cannot be undone. Prefer Keep Table(s) unless you intend to destroy metadata.

Browse vs jobs

Use Browse to inspect or clean up Impala objects. Use New Job to materialize results into tables or CSV files.

12. Recommended user journey

  1. Confirm prerequisites — tools on PATH, writable data root, shared tree deployed, SQL directory ready.
  2. Install (onboard) — run /ads_storage/dispatch/onboard.sh; fix PATH if prompted.
  3. Authenticate — kinit then klist.
  4. Start Dispatch — cd to SQL files, run dispatch.
  5. Create a job — New Job (N), complete the form, Preview, Launch.
  6. Monitor execution — watch Overview status and the detail preview.
  7. Review logs and history — View Logs for the active job; History for older finished work.
  8. Browse metadata when needed — verify tables, sizes, or describe schemas; drop only with care.

13. Troubleshooting

Installation

dispatch command not found

Symptom: shell cannot find dispatch.

Likely cause: ~/.local/bin not on PATH, or onboarding not run.

Action: export PATH="$HOME/.local/bin:$PATH"; which dispatch; rerun /ads_storage/dispatch/onboard.sh.

Shared runtime missing / invalid

Symptom: onboarding or launcher reports shared runtime problems.

Likely cause: Release Operator install not completed or runtime symlink broken.

Action: contact Release Operator; do not run pip.

Authentication

Kerberos missing or expiring

Symptom: sidebar missing TTL; launch refused under 5 minutes.

Likely cause: no ticket or short remaining lifetime.

Action: kinit → klist → relaunch, or press K in New Job.

Application startup

Dispatch not set up for this user

Symptom: notification to run onboard.sh.

Likely cause: private .dispatch home missing.

Action: run /ads_storage/dispatch/onboard.sh.

Terminal too small

Symptom: warning about minimum size.

Likely cause: terminal smaller than 80×24.

Action: enlarge the terminal window.

SSH idle disconnects

Symptom: session drops while idle.

Likely cause: SSH idle timeout.

Action: connect with ssh -o ServerAliveInterval=30.

Job validation

Illegal combination / identifier / email errors

Symptom: validation summary shows issues; Launch disabled or refused.

Likely cause: illegal Source×Destination, bad identifiers, EID table rules, or invalid email.

Action: fix fields until ✓ Ready to launch; consult the legal-cells matrix (M).

Concurrency cap

Symptom: “At the 2-Job concurrency cap…”

Likely cause: two Pending/Running jobs already exist.

Action: wait for one to finish or cancel a running job from Overview.

Job execution

Job Failed with classified code

Symptom: Failed state with SYNTAX / TABLE_NOT_FOUND / MEMORY / AUTH / QUEUE.

Likely cause: matching orchestrator log pattern.

Action: follow the on-screen suggestion (Preview SQL, verify tables, renew Kerberos, retry off-peak, etc.).

Logs

View Logs does nothing useful

Symptom: warning to select a job first.

Likely cause: no row selected on Overview/History.

Action: select a job, then open View Logs.

Browse and Impala metadata

SHOW TABLES failed

Symptom: error notification after Load Tables.

Likely cause: Impala connectivity, Kerberos, or invalid schema/filter.

Action: confirm Kerberos, correct schema/filter, retry Load Tables [S].

Drop confirmation rejected

Symptom: cannot proceed with drop.

Likely cause: confirmation text not exactly I AM SURE.

Action: type the exact phrase, then confirm DROP—or cancel to keep tables.

14. Quick reference

Essential commands

/ads_storage/dispatch/onboard.sh
export PATH="$HOME/.local/bin:$PATH"
which dispatch
kinit
klist
cd /path/to/your/sql/files
dispatch

Core workflow

Onboard → kinit → cd to SQL → dispatch → New Job → monitor Overview → View Logs → History / Browse as needed → Q to quit.

Tab-by-tab purpose

  • Overview — live supervision (7 days)
  • New Job — configure and launch
  • View Logs — full job log / detail
  • History — finished jobs older than 7 days
  • Browse — Impala metadata only

Key warnings

  • Do not run pip or personal venvs for analyst setup.
  • Do not launch with Kerberos missing or under 5 minutes TTL.
  • At most two concurrent Pending/Running jobs.
  • CSV paths stay inside the launch directory.
  • Browse Drop requires I AM SURE and cannot be undone.
